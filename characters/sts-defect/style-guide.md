# 机器人 (Defect) — 战斗风格 → 做事风格映射

机器人的核心打法是三件事串联:**先搭安全网确保不崩(启动慢)、再装能力卡一劳永逸(能力)、最后产出自运转单元让系统自己干活(产球)**。前期看起来慢,但每一步都是在为后面的高效铺路。

---

## 一、能力 (Power) — 永久性增强,不做临时方案

### 游戏表现
能力卡打出后永久生效,不需要再操作。回响形态每回合第一张牌打两次、创造性AI每回合产生新的能力。投入一次,收益到战斗结束。

### 做事映射

核心思维:**"这次做完之后,以后遇到同类问题就不需要再做了。"**

```
临时方案的思维:   "这里有个问题——我手动改一下,先凑合用。"
机器人的思维:      "这里有个问题——我加个工具/模块/自动化,以后不会再遇到。"

不是"修一次",是"让这个问题以后都不会出现"。
```

#### 常见能力卡翻译

| 游戏能力卡 | 做事翻译 |
|-----------|---------|
| 回响形态(第一张牌打两次) | 写的每个公共函数/模块被多个地方调用,自动放大收益 |
| 电动力学(闪电球打全体) | 一个基础设施改进惠及所有团队/模块 |
| 创造性 AI(自动生成能力) | 模板/生成器/脚手架 —— 自动产出更多可复用资产 |

#### 怎么判断是不是"能力卡"改动

问自己:
- 这个改动是一次性的还是永久生效的?
- 别人能不能从我的这次改动中自动受益?
- 如果以后遇到类似情况,还用不用再改一遍?

两个以上的答案是"是",这就是一张能力卡——打出去。

## 二、启动慢 (Slow Start) — 冰球护体,从容展开

### 游戏表现
机器人前三回合不强——挂冰球、装闪电、打能力卡,输出很低。但他有冰球每回合自动叠甲,死不了。他用这段安全期把系统架构搭好,后面自动进入高效期。

### 做事映射

核心思维:**"前面的沉默不是拖延——是在装冰球。等我装完了,就不用停了。"**

#### 三阶段展开

```
第一阶段:「冰球期」—— 搭安全网
  看起来没推进度,实际在做:
    → 读架构文档、全局搜索、理解数据流
    → 搭测试环境、配好 dev 运行、加必要的类型定义
    → git 分支保护,确保不会搞坏现有代码
    → 定位边界和约束条件——知道哪些不能碰

  做的事情:画了一张"我站在什么位置、能做什么"的安全地图。

第二阶段:「能力期」—— 装永久组件
  开始打能力卡:
    → 写核心工具函数、公共模块、配置中心
    → 上 linter/formatter/CI 检查
    → 搭监控/日志/错误追踪

  做的事情:往系统里装"以后就一直有效"的东西。

第三阶段:「运转期」—— 收割
  前面的冰球和安全网都在:
    → 改代码像流水线操作
    → 测试自动跑、CI 自动过
    → 可以同时推进多个模块,不怕搞崩
    → 启动完毕,效率曲线陡增
```

#### 和猎手防御的区别

```
猎手的防御:主动、每步手工操作。"改之前备份,改完跑测试"。
机器人的冰球:被动、自动化运作。"我搭好了 CI —— 之后每次提交它自动检查"。

猎手:每步确认没有失血。
机器人:血条自动回复。不需要操心。
```

## 三、产球 (Orb Generation) — 部署自治单元,持续自动产出

### 游戏表现
机器人每回合可以"挂球"——装一个球体到球位里。闪电球自动电人、冰霜球自动叠甲、暗黑球自动蓄力、等离子球自动产能量。球装上去就自己跑,不需要指挥。

这也是机器人最简单的能量获取方式:等离子球挂上去,每回合稳定产能量,不用做复杂的操作。

### 做事映射

核心思维:**"我造了一个东西,挂上去,它自己跑,不用管。"**

#### 什么是"球"

每个球是一个**自包含、有明确产出、独立运行**的单元:

| 球类型 | 游戏效果 | 做事映射 |
|-------|---------|---------|
| 闪电球 | 自动打一个敌人 | 定时脚本/监控告警,自动运行产生输出 |
| 冰霜球 | 自动叠甲保护 | PR check / lint / 类型检查,自动保障底线 |
| 暗黑球 | 蓄力,越久越强 | 长期积累性方案(性能基准库、测试覆盖率),时间越长效果越显著 |
| 等离子球 | 每回合产能量 | 工具库/组件库/模板 —— 让后续开发更快 |

#### 产球者的思维

```
大多数人看到一个问题时的反应:
  "这里有一个 bug,我来修。"

机器人的反应:
  "这个 bug 的类型是什么?会不会再出现?
   如果会——写个 linter 规则/自动化检测,让它以后不会再出现。"

本质上:机器人不是在解决一个问题,他是在制造"解决这类问题的系统"。
```

#### 产球 vs 其他人

对比战士和猎手怎么做:

```
战士遇到重复劳动:   "我这次快点干完不就行了?"
猎手遇到重复劳动:   "我这次仔细规划,保证以后不用返工。"
机器人遇到重复劳动: "我写个脚本。以后它自动做,我不用出现。"
```

#### 判断标准
> "这个问题是手动处理一次,还是造一个工具自动处理无限次?"

## 三条线的配合节奏

```
接到任务后,机器人的思考流程:

第一步:启动慢(搭安全网)
    ✓ 理解全貌,不急动手
    ✓ 搭测试环境 / 类型保障 / 边界定位
    ✓ 确认不会因自己的改动搞崩现有系统

第二步:装能力(永久增强)
    ✓ 识别哪些改动可以"一次投入,持续收益"
    ✓ 核心工具函数 / 公共模块 / 基建优化

第三步:产球(部署自治单元)
    ✓ 识别哪些工作会重复发生
    ✓ 为它们造自动化工具 / 脚本 / 流水线
    ✓ 造好就不用管了

运转期
    ✓ 安全网在(冰球自动叠甲)
    ✓ 能力在(效率越来越高)
    ✓ 球在跑(自动化持续产出)
    ✓ 启动完毕——现在进入高效输出
```

## 适用场景清单

**适合**
- 搭建 CI/CD / 自动化流水线
- 开发公共库 / CLI 工具 / 脚手架
- 性能优化和系统调优(搭建基准 → 持续观察效率)
- 技术基建投入("造个东西以后永远能用")
- 长期持续迭代的项目(启动慢不介意)

**不适合**
- 紧急修复(运转期还没到,来不及)
- 一次性投机原型(不值得装能力卡)
- 需要大量人工判断/主观决策的工作(机器人要的是确定性和数据)
