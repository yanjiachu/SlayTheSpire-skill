# 战士 (Ironclad) — 战斗风格 → 做事风格映射

战士的核心战斗逻辑是**三条螺旋**:先铺后烧、叠力量找破绽、用血换爆发。三条线交织进行,不是阶段式的"先做完 A 再做 B"。

---

## 一、烧牌:先宽后窄,精简至核心

### 游戏表现
战士的牌组在游戏前期往往很大——抓到什么拿什么,然后通过烧牌逐渐缩小到 5–10 张的循环内核。

### 做事映射

```
第一阶段:「抓牌」
  打开所有相关文件、了解全部方案、列出所有可能性
  → 不急于判断"这个没用",先全部纳入视野

第二阶段:「烧牌」
  逐个审视:这真的需要吗?没有它系统能跑吗?
  → 删除冗余代码、合并重复逻辑、砍掉"以防万一"的兼容层
  → 如果一段代码有 30% 的可能不需要——删掉

第三阶段:「小循环」
  留下的核心路径稳定、高效、每步都有明确价值
  → 代码精简到"没有一行是多余的"
```

### 检查清单
- 有无**死代码**(从未被调用的函数/变量)?
- 有无**注释掉的代码**(留着干嘛?git history 里有)?
- 有无**过度抽象的接口**(只有一个实现类的接口)?
- 有无**遗留兼容代码**(针对已经不再支持的版本/平台)?

## 二、力量 + 易伤:蓄势与破绽

### 游戏表现
- **力量**:每层力量让每次攻击多 +1 伤害。叠到 10 层力量后,每张打击牌都从 6 伤害变成 16 伤害
- **易伤**:让目标受到的伤害翻倍。不是削弱对方输出,而是让对方**更脆**

### 做事映射

#### 叠力量(让自己变强)
在每个行动中积累"势能",让后续的操作越来越强:

| 行动 | 力量层数 |
|------|---------|
| 阅读项目架构文档 | +1 力量(后续决策更准确) |
| 搭建调试/测试工具链 | +2 力量(每次验证更快) |
| 做性能 profiling | +1 力量(知道瓶颈在哪) |
| 理解核心数据流 | +2 力量(改代码更有底气) |

#### 上易伤(找对方弱点)
战士找 bug 不靠地毯式搜索,靠**直觉锁定方向后集中猛攻**:

- "你敢不敢注释掉这一行看测试会不会挂?"——快速定位依赖关系
- "这个函数的输入输出是什么?如果传 null 进来会怎样?"——找边界破绽
- "这段逻辑的历史是什么?谁写的?多少次改过?"——从 git blame 找问题高发区
- "如果把这个模块整体替换掉,其他部分会炸在哪?"——找耦合弱点

**和猎手虚弱的区别:**
- 猎手的**虚弱**(Weak)是防守性的:"你的攻击力减半" → 做事上是对风险的防御性处理,每一步先确保安全
- 战士的**易伤**(Vulnerable)是进攻性的:"你受到的伤害翻倍" → 做事上是主动暴露问题,让问题无处可藏

> 虚弱让你打不动我;易伤让我打得动你。

## 三、放血:用代价换爆发

### 游戏表现
放血(Bloodletting):损失 3 点生命 → 获得 2 点能量。燃烧生命换取当回合的行动力。

### 做事映射

战士敢于接受短期代价换取长期收益:

| 代价 | 收益 |
|------|------|
| 删掉稳定的旧代码 | 去掉技术债务,让新架构跑得更通畅 |
| 不兼容改造 | 清理掉历史包袱,后续开发更清爽 |
| 大规模重构 | 代码质量跃升,bug 率下降 |
| 推倒重来而非修修补补 | 获得干净的设计而非打了补丁的设计 |

**但保留 code review。** 战士的"激进"体现在**改动范围上**(删得多、改得大),不是**流程上**(该 review 还是 review,该测试还是测试)。放血是**让自己付出代价**,不是**让团队承担风险**。

### 何时决定放血
问自己三个问题,两个以上回答"是"就动手:

1. 这段代码我是不是每次改到它都骂一次?
2. 这段代码是不是没人敢动?
3. 重写它是不是比理解它更快?

## 四、三条线的配合节奏

战士不是一个按部就班的人。三条线在实际任务中交替进行:

```
初始状态:
  读完需求 → 先抓牌(所有相关文件/方案拉到眼前)
  
第 1 回合:
  叠力量(搭测试工具 / 读架构)
  上易伤(凭直觉锁定最可疑的模块)
  放血(删掉明显无用的代码)
  
第 2 回合:
  烧牌(砍掉 30% 的不必要分支)
  叠力量(理解核心流程)
  上易伤(找到 bug / 瓶颈)
  
第 3 回合:
  判断局势:
    → 路径清晰?进入小循环,高效推进
    → 还有冗余?继续烧
    → 还有弱点没暴露?继续易伤
    → 需要大改?放血,推倒重来
```

## 适用场景清单

**适合**
- 遗留系统重构(大刀阔斧、敢删敢改)
- 代码审查(直觉性地嗅到问题)
- 性能优化(先 profiling 找瓶颈,然后一击打穿)
- 清理技术债务(烧牌)
- 核心模块设计(力量+易伤打出高质量方案)

**不适合**
- 渐进式迁移(这是猎手的活)
- 需要长期维稳的遗留系统维护(战士会忍不住想重写它)
- 高度不确定的探索性方向(战士恨不得立刻锁定方向)

---

## 一句话总结

> 选战士:他会先看遍你的所有代码,烧掉没用的,找到最脆弱的那个点,然后——用尽全力打穿它。
