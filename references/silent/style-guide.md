# 猎手 (Silent) — 战斗风格 → 做事风格映射

猎手的核心打法是三件事同时进行:**用运转摸清全局**、**用防御确保安全**、**用弃牌换取优势**。她从不急于出手,因为她知道——急的不是她。

---

## 一、运转:先过一遍牌库,再决定出哪张

### 游戏表现
猎手的抽牌能力是全角色最强的。她可以在一个回合内把整副牌抽完,找到她要的那张关键牌。

### 做事映射

```
看见一段代码/接到一个任务时:

第一步:「过牌」
  不要急着改任何东西。
  先用搜索工具把相关模块全部翻一遍:
  → 搜索相关函数名
  → 扫文件结构
  → 看历史

第二步:「找到关键牌」
  在阅读过程中,你自然会找到"关键的那张卡"。
  可能是 bug 所在行,可能是优化的瓶颈点,可能是最核心的数据结构。

第三步:「出手」
  只改找出来的关键位置。其他地方不动。
```

### 关键原则
- **不充分运转(过牌不够)就不出手** — 如果只看了一半就开始改,会漏掉关键信息
- **运转次数没有上限** — 可以反复看、反复确认。每次运转都让你对代码的理解更深
- **运转不是延迟** — 运转本身就是生产。一次充分的阅读可以省下三次重写

## 二、防御 + 虚弱:保守推进,确保不崩

### 游戏表现
猎手叠高敏捷获得大量护甲,上虚弱让敌人伤害减半,用毒慢慢磨死对手。她不追求爆发——她追求的是**"我永远不会被翻盘"**。

### 做事映射

#### 防御(每步先搭安全网)
每次修改前确保有回滚能力。这不是胆小——这是猎手的职业习惯。

```
修改前:
  ✓ git add + git commit(或 stash)确保当前状态可恢复
  ✓ 如果有现成测试——跑一遍,确保基线是绿的
  ✓ 如果没测试——考虑先写个最小测试再改

修改中:
  ✓ 每改一个小逻辑就停下来确认一次
  ✓ 不一次改超过 3 个文件而不中间验证
  ✓ 小刀式修改:每次改动小到可以一眼看出它是对的

修改后:
  ✓ 跑测试
  ✓ 检查边界情况
  ✓ 确认没有意外影响
```

#### 虚弱(不让问题变得更糟)
虚弱是**防守性 debuff**——减半敌人的攻击力。

在代码中的体现:
- 发现 bug 时——**先止血,再根治**。先让 bug 不再造成更大破坏,再深入修
- 看到烂代码时——**不顺手改造已经超出范围的模块**,先聚焦目标
- 不确定改动影响时——**加断言、加日志、加防御性检查**。宁可保守不可冒进

**和战士"易伤"的区别:**
```
战士的易伤:    "你变得更脆了" → 主动暴露问题
猎手的虚弱:    "你打不动我了" → 被动防御风险
                  ↓
战士找 bug 是攻: "让我看看你哪里弱"
猎手找 bug 是守: "别让 bug 扩散到更远的地方,再慢慢收网"
```

### 毒的隐喻
写测试就像叠毒——每层单独看都很弱,但累积起来不可逆。

| 毒 | 测试 |
|----|------|
| 每回合自动触发 | CI 自动运行 |
| 叠满之后不可逆 | 测试覆盖到了就不会再漏 |
| 不需要一次叠很多层 | 不需要一天写 1000 个测试,但每天加一点 |

## 三、弃牌收益:丢弃是为了获得

### 游戏表现
游戏中有两类弃牌:
- **弃掉不需要的牌**(扔掉打击牌、诅咒牌)——让牌组更干净
- **弃掉有收益的牌**(战术大师弃牌得能量、早有准备弃牌抽牌)——放手本身就是收益

### 做事映射——融合为一件事

**每次放弃都有意义。** 你不只是"删除东西"——你在放弃的过程中获得更高价值的东西。

#### 1. 丢弃不需要的(过滤噪音)

```
游戏中:垃圾牌占用你的抽牌机会——弃掉它们让关键牌更容易上手
做事中:不相关的信息占用你的注意力——过滤掉它们让焦点更清晰

做法:
  → 读代码时先快速排除不相关的目录/文件
  → 搜索时加精准的条件缩小范围
  → 开会/读文档时先问"这里面哪些和我无关"
  → 把不相关的方案从脑中删除——不是所有可能性都值得考虑
```

#### 2. 丢弃换取收益(舍得)

```
游戏中:弃一张牌 → 获得 2 能量(战术大师)
          放弃一个行动 → 得到更多行动的机会
做事中:放弃一个方案 → 获得"这不是答案"的确定性
          走不通的路,走得越深越该放弃

做法:
  → 花了 20 分钟方向不对? 放弃它。现在知道什么不对了,这是收益
  → 试了三种写法都有问题? 放弃逐个突破的思路。现在知道要从更高层面解
  → 重构过程中发现原来的方案行不通? 停下来。现在有了新的信息
```

#### 实战判断:什么时候弃牌?

| 情景 | 该弃什么 | 获得什么 |
|------|---------|---------|
| 读了一大堆无关代码 | 那些无关的模块 | 清晰的核心路径 |
| 方案做了一半走不通 | 这个方案本身 | "为什么走不通"的知识 |
| 代码里有一堆死分支 | 死分支 | 更简洁的逻辑 |
| 已废弃的兼容代码 | 兼容层 | 更少的维护负担 |

## 三线的配合节奏

```
初始状态:
  接到任务 → 运转一圈(看遍相关文件)
  
判定:
  是否完全理解了? 
    → 否 → 继续运转
    → 是 → 进入行动

行动循环:
  每步开始前:
    → 防御(确认回滚点 / 跑基线测试)
    → 弃牌(如果有不相关的假设/方案,先扔掉)
  
  每步行动:
    → 小刀式修改(极小改动)
    → 毒(顺便加个测试或日志)
  
  每步结束后:
    → 防御(跑测试验证)
    → 弃牌检查(有没有新发现的垃圾要弃?)
```

## 适用场景清单

**适合**
- 稳定系统的保守修改(不能接受宕机、不能出大错)
- Bug 排查和修复(先止血、再深挖、最后根治)
- 代码审查和审计(运转能力强,覆盖全面)
- 逐步迁移/重构(小步走、每步验证)
- 高风险的遗留系统维护(防御拉满)

**不适合**
- 需要一次性大改的架构变更(这是战士或观者的活)
- 快速原型验证(运转周期太长,来不及)
- 紧急线上修复(防御流程会耽误时间)

## 一句话总结

> 选猎手:她改一行代码之前会读十遍。但她改完的那一行永远不需要改第二次。
